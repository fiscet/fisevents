# UX Analysis & Improvement Plan: Event Attendant Management

## Current State Analysis

After reviewing the codebase, specifically the creator-admin event management features (`EventAttentantTable`, `EventAttentantCards`) and the underlying backend actions (`lib/actions.ts`), here is the current state of the UX for managing event attendants:

- **View-Only:** The creator admin panel only provides a read-only list of attendants.
- **Visible Data:** It displays the attendant's Full Name, Email, Phone, and Subscription Date.
- **Missing Features in Admin:** The backend actions for adding (`addEventAttendant`) and removing (`removeEventAttendant`) exist, but they are currently only utilized by the public-facing subscription and unsubscription flows (`app/[lang]/pe/`). They are not exposed to the event creator.

## UX Gaps for Target Users

For users like a yoga studio owner or a restaurant owner organizing a party, the current UX is insufficient. These types of events require hands-on management:

1.  **Manual Additions:** Owners frequently take reservations over the phone or have walk-ins. There is currently no way for an admin to manually add an attendant to the list.
2.  **Manual Cancellations:** If an attendant calls to cancel, the admin cannot remove them from the list to free up space (`maxSubscribers`).
3.  **Check-in Management:** At the door of the event or studio, the owner needs to check people off as they arrive to track actual attendance versus registrations.
4.  **Payment Tracking:** For events where payment is taken at the door (e.g., cash for a yoga class), there is no way to track who has paid.
5.  **Offline Access (Export):** Owners often prefer to print out a physical list to use at the door. There is no export functionality.

## Proposed Improvement Plan

To make the application viable for these use cases, we need to implement the following improvements:

### Phase 1: Basic Management (CRUD)

- **Implement "Add Attendant" in Admin:** Create a button and a modal form in the `EventAttentantList` view allowing the admin to manually add an attendant. This will utilize the existing `addEventAttendant` action.
- **Implement "Remove Attendant" in Admin:** Add a delete action (e.g., a trash icon) to each row/card in the attendant list, with a confirmation dialog. This will utilize the existing `removeEventAttendant` action.

### Phase 2: Status & Tracking

- **Schema Update (Next.js & Sanity Studio):**
  - Extend the `EventAttendant` type in the Next.js app (`fisevents/types/sanity.types.ts`).
  - Update the actual Sanity schema in the separate Studio directory (`studio/schemaTypes/components/eventAttendant.ts`) to include:
    - `checkedIn` (boolean, default false)
    - `paymentStatus` (string: 'pending', 'paid', default 'pending' - if applicable)
- **UI Updates:**
  - Add columns/indicators in `EventAttentantTable` and `EventAttentantCards` for the new statuses.
  - Add quick-action toggles directly in the list (e.g., a checkbox to quickly mark someone as "Checked In").

### Phase 3: Utility Features

- **CSV Export:** Add an "Export to CSV" button that generates a downloadable file containing the attendant list, which is highly requested for door management.

### Phase 4: Authentication Simplification

- **Remove Google Login Config:** Remove the `GoogleProvider` from the NextAuth configuration in `lib/auth.ts`, leaving only the `EmailProvider`.
- **Clean Up Auth UI:**
  - Remove the Google login option from `SignInProviders.tsx`.
  - Delete `SignInWithGoogle.tsx`, `GoogleLogo.tsx`, and their associated story files since they will no longer be needed.
  - Ensure the auth page layout looks good with only the Email Magic Link option.
